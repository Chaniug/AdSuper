## 用户需求

修复 `sync_issues.py` 中的 `extract_rules_from_issue` 函数，使其能完整提取所有 adblock 规则格式，包括高级过滤规则（如属性选择器、复杂 CSS 选择器、响应头规则等）。

## 当前问题

1. **正则表达式不完整**：当前使用的正则表达式模式复杂且有限制性，可能遗漏某些有效的规则格式
2. **中文字符干扰**：正则表达式中包含中文字符（如 `，。！？；：、…`），难以维护且可能导致匹配错误
3. **高级规则不支持**：无法正确提取高级过滤规则，如 `##^responseheader:`，`$removeparam=`，`:has()` 伪类等

## 预期目标

- 能够提取所有常见的 adblock 规则格式
- 支持高级过滤规则（uBlock Origin 扩展语法）
- 保持向后兼容性，不影响现有功能

## 技术方案

### 实施策略

重写 `extract_rules_from_issue` 函数，采用**两步验证法**：

1. **粗提取**：使用简单模式识别可能的规则行（减少误判）
2. **精验证**：使用 `RuleValidator.validate_rule()` 方法验证提取的内容是否确实是规则

### 关键决策

1. **复用验证逻辑**：不在 `sync_issues.py` 中维护独立的正则表达式，而是复用 `scripts/rule_validator.py` 中已修复的验证模式
2. **代码块识别**：优先从 Markdown 代码块（``` 包围）中提取规则，减少误判
3. **宽松提取**：对于非代码块内容，采用宽松的匹配策略，然后使用验证器过滤

### 实施步骤

#### 1. 修改 `extract_rules_from_issue` 函数

- 创建 `RuleValidator` 实例
- 识别 Markdown 代码块（``` 标记）
- 在代码块中：提取所有非空行，使用 `validator.validate_rule()` 验证
- 在代码块外：使用简单模式预过滤，然后使用 `validator.validate_rule()` 验证

#### 2. 更新 `rule_validator.py`（如需要）

- 确保正则表达式支持所有高级规则格式
- 添加对 `##^responseheader:`，`$removeparam=` 等高级语法的支持

#### 3. 测试

- 使用实际 issue 内容测试规则提取
- 确保能提取所有常见的规则格式

### 性能考虑

- 正则表达式预编译（已在 `RuleValidator` 中完成）
- 避免重复验证（使用集合去重）

### 可靠性保障

- 向后兼容：现有 issue 规则应能被正确提取
- 错误处理：验证失败的规则应记录日志，但不中断流程
